Git Commands

"Think of Git as a time machine for your code."

20 Commands 4 Scenarios Step-By-Step Beginner → Advanced
Setup & Configuration
Command 01
Check Git Version
Config

Verify Git is installed on your system and check which version you're running. Always a good first step before working on a new machine.

Run from: Anywhere in terminal
bash
git --version
Expected Output
git version 2.49.0 — any 2.x or higher is fine.
Command 02
Configure Username & Email
Config · First Time Only

Git needs to know who you are before it records any commits. This is a one-time global setup — your name and email appear in every commit you make across all repos.

Run from: Anywhere (sets global config)
bash
# Set your display name git config --global user.name "Krithika" # Set your email (match your GitHub account) git config --global user.email "yourmail@gmail.com" # Verify everything looks right git config --list
Why it matters
Every commit is stamped with this identity. On team projects, teammates see your name in git log.
Creating a Repository
Command 03
git init — Start Tracking a Folder
Essential

Initializes a brand-new Git repository in the current folder. Git creates a hidden .git/ subfolder that silently stores your entire version history.

Run from: Inside your project folder
bash
# 1. Create project folder and enter it mkdir my-project cd my-project # 2. Initialize Git here git init
Output you'll see
Initialized empty Git repository in /my-project/.git/ — that .git/ folder is now your time machine.
Status, Staging & Committing
Command 04
git status — What's Going On?
Most Used

Your most-used command. Shows which files are modified, which are staged for commit, which are new (untracked), and which have been deleted. Run this constantly — it's always safe.

Run from: Inside any Git repo
bash
git status
What you'll see
Untracked files: app.py — meaning Git sees the file but isn't tracking it yet. Red = not staged. Green = staged and ready to commit.
Command 05
git add — Stage Your Changes
Essential

Moves files into the staging area — a holding zone where you prepare exactly what goes into the next snapshot. Think of it as packing a box before shipping it.

Run from: Inside the Git repo
bash
# Stage a single file git add app.py # Stage multiple specific files git add file1.py file2.py # Stage EVERYTHING changed (most common) git add .
Pro tip
Use git add . when you're sure about all changes. Use specific filenames to be selective — great for keeping unfinished work out of a commit.
Command 06
git commit — Save a Snapshot
Essential

A commit is a permanent snapshot of your staged changes. The message (-m) is your note to future-you (and teammates) explaining what changed and why.

Run from: Inside the Git repo, after staging
bash
# Standard commit with a message git commit -m "Added login feature" # First-ever commit convention git commit -m "Initial project setup"
💡 Write commit messages in present tense: "Add login" not "Added login". It reads like "If applied, this commit will Add login."
Viewing History
Command 07
git log — Browse the Timeline
Essential

Shows the full commit history with author, date, and message for every snapshot ever made. Use the --oneline flag for a clean, condensed view that fits on screen.

Run from: Inside any Git repo
bash
# Full detailed history git log # Compact one-line view (preferred for quick scanning) git log --oneline
--oneline output looks like
a1b2c3 Added login  ·  d4e5f6 Initial commit — each line is a commit hash + your message.
GitHub Remote & Pushing
Command 08
Connect Local Repo to GitHub
Remote

Links your local Git repo to a remote repository on GitHub. "origin" is just a nickname for the URL — the convention everyone uses. Do this once per project.

Run from: Inside your project folder, after git init
bash
# Add remote (replace URL with your repo) git remote add origin https://github.com/Krithiikaa/my-project.git # Verify it was added correctly git remote -v
Before running this
Create the repository on GitHub first (browser), then copy the .git URL from the green Code button.
Command 09–10
git push — Upload to GitHub
Remote

Sends your local commits to GitHub. The first time you push, use -u origin main to set up tracking. After that, plain git push is enough.

Run from: Inside the repo, after committing
bash
# First push — sets upstream tracking git push -u origin main # If your branch is called "master" instead git push -u origin master # Every push after the first git push
Clone, Pull & Fetch
Command 11
git clone — Download a Repo
Remote

Creates a full local copy of any GitHub repository. Better than downloading a ZIP — you get the entire history and can push changes back.

Run from: The parent folder where you want the project
bash
# General form git clone https://github.com/user/project.git # Real example — your hiring app git clone https://github.com/Krithiikaa/predictive-hiring-app.git
What happens
Git creates a folder named predictive-hiring-app/ with all the code and history inside. cd into it and start working.
Command 12–13
git pull & git fetch — Stay Updated
Team

git pull downloads your teammate's latest changes AND merges them into your branch — it's a fetch + merge in one shot. git fetch only downloads, letting you review before merging.

Run from: Inside the repo on the branch you want to update
bash
# Pull = fetch + merge (most common) git pull # Fetch only — download but don't merge yet git fetch # After fetch, you can compare before merging git merge origin/main
⚡ Always git pull before starting new work on a team project. This avoids nasty merge conflicts.
Branching & Merging
Command 14
git branch — Create & View Branches
Branching

Branches are parallel versions of your code. Create a new branch for each feature or bugfix — it keeps main clean and your work isolated until it's ready.

Run from: Inside the repo
bash
# Create a new branch git branch login-feature # List all branches (* = current) git branch # Output: # * main # login-feature
Command 15
git switch / checkout — Change Branch
Branching

Switches your working directory to a different branch. git switch is the modern way (Git 2.23+); git checkout is the classic method that still works everywhere.

Run from: Inside the repo
bash
# Modern way (preferred) git switch login-feature # Classic way (still works) git checkout login-feature # Create AND switch in one command git switch -c chatbot-feature git checkout -b chatbot-feature # same thing
Command 16–17
git merge & git branch -d — Merge & Clean Up
Branching

Merge brings a branch's commits into your current branch (usually main). After merging, delete the branch to keep things tidy.

Run from: The branch you want to merge INTO (usually main)
bash
# Step 1: Go to main git switch main # Step 2: Bring in the feature branch git merge login-feature # Step 3: Delete the branch (work is done) git branch -d login-feature
Undoing Mistakes
Command 18
git restore — Unstage or Discard
Undo

Two rescue operations in one command. Unstage a file you added by mistake, or throw away local edits entirely and restore the last committed version.

Run from: Inside the repo
bash
# Remove from staging (file stays edited) git restore --staged app.py # Throw away local changes (⚠️ permanent!) git restore app.py
⚠️ git restore app.py (without --staged) is destructive — your unsaved edits are gone. Always double-check which form you need.
Command 19
git reset — Roll Back Commits
Undo · Dangerous

Moves the branch pointer backwards in history. --soft keeps your file changes (just undoes the commit). --hard nukes both the commit and all changes. Use hard with extreme caution.

Run from: Inside the repo
bash
# Undo last commit, keep file changes git reset --soft HEAD~1 # Undo last commit AND delete all changes (dangerous!) git reset --hard HEAD~1 # HEAD~2 goes back 2 commits, HEAD~3 goes back 3, etc.
Command 20
.gitignore — Hide Files from Git
Essential

A plain text file that tells Git which files to never track. Crucial for keeping secrets (API keys, passwords in .env), virtual environments, and compiled cache files out of your repository.

Create in: Root of your project folder
.gitignore
# Python virtual environment venv/ # Secret keys — NEVER commit this .env # Python cache __pycache__/ *.pyc # VS Code settings (optional) .vscode/
For your FastAPI / Python projects
Always add venv/ and .env to .gitignore before your very first commit. Once a secret is committed, it's in history forever.
Stash, Diff & File Operations
Command 21
git stash — Shelve Work Temporarily
Handy

Saves your uncommitted changes to a hidden shelf so you can switch branches without losing your in-progress work. Pop them back when you return.

Run from: Inside the repo when you have uncommitted changes
bash
# Shelve current work git stash # Do other work, switch branches, etc. git switch main # Come back and restore your stashed work git stash pop
Command 22
git diff — See What Changed
Inspect

Shows the exact line-by-line differences between your working files and the last commit (or staged area). Green lines are additions, red are deletions.

Run from: Inside the repo
bash
# Compare working files vs last commit git diff # Compare staged changes vs last commit git diff --staged
Command 23–24
git rm & git mv — Remove or Rename
Files

Git-aware file operations. Using these instead of regular delete/rename means Git automatically stages the change — no need for a separate git add.

Run from: Inside the repo
bash
# Delete a file and stage the deletion git rm app.py git commit -m "Removed old app.py" # Rename a file (Git tracks it as a rename) git mv old_name.py new_name.py
Scenario Workflows

Scenario 1 — Brand New Project

①mkdir project && cd project ②git init ③git add . ④git commit -m "Initial commit" ⑤git remote add origin <repo-url> ⑥git push -u origin main

Scenario 2 — Existing GitHub Project

①git clone <repo-url> ②cd project ③git pull # get latest ④git add . ⑤git commit -m "Updated code" ⑥git push

Scenario 3 — Daily Team Workflow

# — Before starting work — git pull # — After writing code — git add . git commit -m "Implemented API endpoint" git push

Scenario 4 — Feature Branch (Internship Standard)

①git switch -c chatbot-feature ② # ... write code ... ③git add . ④git commit -m "Added chatbot module" ⑤git push origin chatbot-feature ⑥ # Open Pull Request on GitHub ↗
